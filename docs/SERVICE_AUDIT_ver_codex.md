# 서비스화 점검 보고서 ver_codex

작성일: 2026-07-07  
대상: `fable` 퀀트 백테스트 대시보드  
점검 범위: `README.md`, `docs/`, `apps/engine`, `apps/web`, `.github/workflows/ci.yml`

## 1. 한 줄 결론

현재 프로젝트는 개인용 로컬 백테스트 도구로는 완성도가 높고, 특히 "코드 없이 전략 생성"과 "과적합 검증을 기본으로 보여주는 구조"가 강점이다. 다만 외부 사용자에게 서비스하려면 인증, 사용자별 데이터 격리, 합법적 시세 데이터 소스, rate limit, 운영 모니터링, 법적 고지가 먼저 필요하다.

서비스화 관점에서 가장 큰 병목은 기능 부족보다 멀티테넌시와 운영 안정성이다. 차별화는 이미 가진 검증 철학을 신뢰 점수, 페이퍼 트레이딩, AI 전략 코치로 제품화하는 방향이 가장 효율적이다.

## 2. 현재 제품 상태

### 이미 갖춘 강점

- FastAPI 엔진이 백테스트, 최적화, 워크포워드, 몬테카를로, 멀티 심볼 검증, 전략 비교, 시그널 체크를 제공한다.
- Next.js 웹은 홈, 백테스트, 저장 전략, 실전 시그널 흐름을 갖췄고 고급 모드에서 JSON 편집과 검증 도구를 노출한다.
- 전략은 임의 코드가 아니라 Pydantic DSL 스키마를 통과한 JSON만 실행한다. AI 전략 생성과 블록 빌더도 같은 검증 경로를 탄다.
- 백테스트 엔진은 look-ahead 방지를 명시하고 다음 봉 체결, 수수료, 슬리피지, 롱/숏, 포지션 사이징, 장중 손절/익절을 처리한다.
- 과적합 진단이 제품 핵심에 있다. IS/OOS, 워크포워드, 몬테카를로, 멀티 심볼 검증은 일반적인 "수익률만 보여주는" 백테스트 도구와 차별화된다.
- CI는 엔진 pytest와 웹 빌드를 실행하도록 구성되어 있다.

### 현재 구조상 한계

- 저장소는 `apps/engine/store.py`의 단일 SQLite 파일 `storage/strategies.db`를 사용한다. 사용자 개념이 없어 공개 시 모든 사용자가 같은 전략과 감시 목록을 공유하게 된다.
- `apps/engine/main.py`의 엔드포인트는 인증 없이 열려 있다. 저장, 삭제, AI 생성, 최적화 같은 비용성 작업도 보호되지 않는다.
- CORS 허용 오리진이 localhost로 하드코딩되어 있어 배포 환경변수화가 필요하다.
- 주식 데이터는 `yfinance`, 크립토는 `ccxt`/바이낸스에 의존한다. `yfinance`는 상업 서비스의 안정적 데이터 소스로 보기 어렵다.
- 긴 작업은 동기 요청으로 처리된다. 동시 사용자가 생기면 최적화, 워크포워드, 멀티 심볼, AI 생성이 API 워커를 점유할 수 있다.
- 웹 E2E 테스트 파일은 없다. 엔진 단위 테스트는 많지만 사용자 플로우 검증은 비어 있다.

## 3. 서비스 출시 전 필수 보완

### P0. 계정, 권한, 데이터 격리

현재 구조로는 외부 공개가 어렵다.

- 이메일 매직링크, OAuth, 또는 간단한 세션 기반 로그인 도입
- `strategies`, `watches`에 `user_id` 추가
- 모든 저장/조회/삭제 API에서 소유자 검증
- SQLite에서 PostgreSQL로 전환
- DB 마이그레이션 도구 도입
- 계정 삭제, 데이터 내보내기 기능 준비

최소 출시 기준은 "내 전략과 감시 목록은 나만 보고 수정할 수 있다"이다.

### P0. 시세 데이터 소스 정리

가장 큰 서비스 리스크다.

- `yfinance`는 로컬 실험에는 적합하지만 외부 유료/상업 서비스의 기본 데이터 소스로는 불안정하다.
- 대안 후보: Polygon, Twelve Data, Alpha Vantage, 한국투자증권 OpenAPI, KRX 데이터, Nasdaq Data Link 등
- 국내 주식 타겟이면 KRX 종목명 검색, 상폐 포함 데이터, 국내 수수료/세금 프리셋까지 같이 설계해야 한다.
- 크립토는 현행 `ccxt`를 유지하되 거래소별 장애, rate limit, 심볼 지원 차이를 UI에 안내해야 한다.

데이터 API 비용은 곧 서비스 원가이므로 요금제와 함께 결정해야 한다.

### P0. API 보호와 남용 방지

현재 무인증 API는 서비스 운영에 취약하다.

- 엔진 API 인증 미들웨어 추가
- IP/사용자별 rate limit
- `/ai/generate` 일일 쿼터
- `/optimize`, `/walkforward`, `/multisymbol` 요청 한도와 타임아웃
- 요청 본문 크기 제한
- 공개 API 경로는 `/v1`처럼 버저닝
- CORS 허용 오리진 환경변수화

특히 최적화는 최대 300조합, 멀티 심볼은 최대 8종목, 몬테카를로는 최대 5000회라 익명 호출을 열어두면 비용과 성능 문제가 바로 발생한다.

### P0. 법적 고지와 서비스 약관

이 프로젝트는 시그널과 백테스트 결과를 제공하므로 고지가 중요하다.

- 모든 결과 화면에 "투자 자문이 아니며 과거 성과는 미래 수익을 보장하지 않는다"는 문구 표시
- 시그널 제공 방식이 유사투자자문업 신고 대상인지 검토
- 이용약관, 개인정보처리방침, 환불/면책 조항 작성
- 생존 편향, 데이터 지연, 데이터 오류 가능성 안내
- 자동매매나 브로커 연동을 추가할 경우 별도 리스크 고지

현재 시그널 페이지에는 기본 문구가 있으나 백테스트 결과, 전략 공유, 리포트 내보내기까지 일관되게 들어가야 한다.

### P0. 배포와 운영 인프라

권장 구조:

- 웹: Vercel
- 엔진: Cloud Run, Fly.io, Railway, Render 중 하나의 컨테이너 환경
- DB: 관리형 PostgreSQL
- 캐시: 볼륨, Redis, 또는 object storage
- 비동기 작업: Celery, RQ, arq, 또는 Cloud Tasks
- 모니터링: Sentry, structured logging, health check, uptime monitor
- 백업: DB 자동 백업과 복구 리허설

컨테이너 로컬 디스크에 시세 캐시와 SQLite를 두는 구조는 배포 후 인스턴스 재시작, 스케일아웃, 백업에서 문제가 된다.

## 4. 차별성을 키울 기능

### 1. 전략 신뢰 점수

현재 엔진은 IS/OOS, 워크포워드, 몬테카를로, 멀티 심볼 검증 결과를 이미 만든다. 이를 하나의 종합 등급으로 묶으면 제품 정체성이 선명해진다.

예:

- A: OOS 양호, 워크포워드 양호, 몬테카를로 손실확률 낮음, 여러 종목에서 일관
- B: 전체 성과는 좋지만 일부 검증 약함
- C: 수익은 있으나 과적합 의심
- D/F: 거래 수 부족, OOS 붕괴, 손실확률 높음

"수익률 300%"가 아니라 "신뢰 등급 B"로 말하는 서비스가 되면 차별화가 크다.

### 2. AI 전략 코치

현재 AI는 자연어를 DSL로 바꾸는 역할이다. 다음 단계는 백테스트 결과를 읽고 개선안을 제안하는 코치다.

기능 흐름:

1. 사용자가 전략을 실행한다.
2. 엔진 결과와 과적합 진단을 AI에 전달한다.
3. AI가 약점, 과적합 원인, 수정 후보를 설명한다.
4. 사용자는 "적용" 버튼으로 파라미터나 조건을 바꾼다.

이 기능은 기존 DSL, 결과 JSON, Anthropic 연동 구조를 활용할 수 있어 개발 비용 대비 차별성이 높다.

### 3. 페이퍼 트레이딩

저장 전략을 "추적 시작"하면 그 이후의 시그널만 기록해 진짜 미래 구간 성과를 쌓는 기능이다.

효과:

- 백테스트 과시와 실제 운용 사이의 신뢰 격차를 줄인다.
- 공유 전략의 순위를 과거 백테스트가 아니라 추적 시작 이후 성과로 매길 수 있다.
- 시그널 체크 인프라가 이미 있어 구현 기반이 있다.

서비스의 핵심 메시지를 "과거에 맞춘 전략"이 아니라 "앞으로도 검증되는 전략"으로 바꿀 수 있다.

### 4. 시그널 자동 알림

현재 시그널은 사용자가 수동으로 확인한다. 스케줄러와 알림을 붙이면 유료화 포인트가 된다.

- 매일/매시간 자동 체크
- 이메일, 텔레그램, 카카오 알림
- 진입, 청산, 보유 상태 변화만 알림
- 알림 이력과 missed signal 로그
- 알림별 실제 다음 봉 성과 추적

백테스트 도구에서 운용 보조 도구로 확장되는 가장 자연스러운 기능이다.

### 5. 포트폴리오 백테스트

현재 멀티 심볼 검증은 같은 전략을 여러 종목에 각각 적용하는 구조다. 다음 차별화는 자본 배분 단위의 포트폴리오 검증이다.

- 여러 전략과 여러 종목 조합
- 종목별/전략별 비중
- 월간/분기 리밸런싱
- 상관관계와 분산 효과
- 벤치마크 대비 알파
- 포트폴리오 단위 MDD, Sharpe, Calmar

개별 전략이 아니라 실제 자금 운용에 가까운 분석이 된다.

### 6. 한국 시장 특화

국내 사용자를 타겟으로 할 경우 가장 강한 차별화 포인트다.

- KRX 전 종목 한글 검색
- 코스피/코스닥 필터
- ETF, ETN, 우선주 구분
- 원화 표기
- 증권거래세, 수수료, 슬리피지 프리셋
- 상장폐지/거래정지/액면분할 데이터 처리
- 국내 휴장일 캘린더

지금은 `005930.KS`처럼 티커를 아는 사용자를 전제로 한다. 서비스로 가려면 검색과 데이터 품질이 중요하다.

### 7. 전략 공유와 검증 리더보드

DSL이 JSON이라 임의 코드 실행 없이 안전하게 공유할 수 있다. 이 구조는 커뮤니티 기능에 유리하다.

- 전략 공개 링크
- 전략 포크
- 공유 카드에 신뢰 점수 표시
- 백테스트 기간과 데이터 소스 명시
- OOS 성과만 반영하는 리더보드
- "공개 이후 성과" 기준 대회

단, 공유 기능은 악성 홍보나 투자 권유로 변질될 수 있어 신고, 모더레이션, 법적 고지가 필요하다.

## 5. 개선해야 할 점

### 백테스트 정확도

- 숏 보유 비용 미반영: 주식 대차 수수료, 크립토 펀딩비가 없다.
- 배당 현금흐름 미모델링: `auto_adjust` 수정주가는 반영되지만 배당 재투자 흐름은 별도 모델이 아니다.
- 체결 모델 선택지 부족: 현재 기본은 다음 봉 시가 근사다. 종가 체결, VWAP 근사, 지정가 미체결 가능성 등 옵션이 있으면 더 현실적이다.
- 부분 청산과 피라미딩이 없다.
- ATR 기반 손절/사이징은 지표 기반이 있으므로 DSL risk 확장으로 추가 가능하다.
- 생존 편향 고지가 필요하다. 상장폐지 종목이 데이터에 없으면 장기 백테스트가 낙관적으로 보인다.

### 데이터 품질

- CSV 캐시 손상 감지와 자동 재다운로드가 없다.
- 캐시 용량 상한이 없다.
- 실패한 티커를 반복 조회할 때 부정 캐싱이 없다.
- 주식과 크립토의 타임존 처리가 다르다. 시그널의 "오늘" 판단에서 오차가 생길 수 있다.
- yfinance 1시간봉의 기간 제한 같은 데이터 소스 한계를 UI에서 선제 안내해야 한다.

### 보안

- 인증 없는 저장/삭제 API는 공개 불가다.
- 전략 저장소의 전역 한도 `MAX_STRATEGIES=200`, 감시 한도 `MAX_WATCHES=50`은 사용자별 한도가 아니다.
- DSL 중첩 깊이 제한은 명시적으로 두는 편이 좋다. 현재 Pydantic 검증에 기대는 구조다.
- AI 생성 프롬프트와 응답 로그의 개인정보/민감정보 취급 방침이 필요하다.
- 의존성 취약점 스캔이 CI에 없다.

### UX

- 첫 방문 온보딩이 약하다. 예제 전략과 예제 결과를 바로 보여주면 전환이 좋아진다.
- 결과 상태가 페이지 이동 시 사라진다. URL 파라미터나 서버 저장 결과로 공유 가능한 링크를 만들 수 있다.
- 로딩 스켈레톤이 부족하다. 긴 작업에서는 진행률 또는 예상 시간이 필요하다.
- 모바일 실기기 점검이 필요하다. 테이블, 히트맵, 캔들 차트는 작은 화면에서 깨지기 쉽다.
- 접근성 점검이 필요하다. 키보드 이동, focus ring, 색약 대비, aria label을 보강해야 한다.
- 페이지별 SEO 메타가 부족하다.

### 테스트와 품질

- 엔진 단위 테스트는 풍부하지만 웹 E2E 테스트가 없다.
- API 스모크 테스트는 일부 엔드포인트에 집중되어 있다. `/walkforward`, `/multisymbol`, `/watches`, `/signals/check`도 API 레벨 테스트가 필요하다.
- AI 생성은 실제 API 키 없이 테스트하기 어렵다. mock 기반 회귀 테스트와 샘플 프롬프트 골든 테스트를 분리해야 한다.
- 수치 골든 테스트가 있으면 리팩터링 때 백테스트 결과 변화를 빨리 감지할 수 있다.
- 부하 테스트가 필요하다. 최적화와 워크포워드의 동시 요청 한계를 측정해야 한다.

## 6. 권장 로드맵

### Phase 0. 로컬 제품 완성도

## 8. 점검 중 확인한 검증 상태

- `python -m pytest tests -q`를 실행했다.
- 결과는 99개 중 92개 통과, 7개 에러였다.
- 에러는 `tests/test_loader.py`의 `tmp_path` fixture setup에서 발생한 Windows 임시 디렉터리 접근 권한 문제다.
- 두 번째 실행에서 `TMP`/`TEMP`와 `--basetemp`를 워크스페이스 안으로 지정했지만 동일 계열의 권한 문제로 종료됐다.
- 출력상 코드 assertion 실패가 아니라 테스트 환경의 임시 디렉터리 접근 문제로 판단된다.
- 웹 E2E 테스트는 현재 확인되지 않았다.

## 9. 바로 실행할 10개 작업

1. `SERVICE_AUDIT_ver_codex.md` 기준으로 P0 항목을 이슈로 쪼갠다.
2. 사용자 인증 방식을 결정한다.
3. PostgreSQL 스키마 초안을 만든다.
4. `strategies`, `watches`에 `user_id`를 추가하는 마이그레이션을 설계한다.
5. `NEXT_PUBLIC_ENGINE_URL`, CORS origin, API secret을 환경변수화한다.
6. `/ai/generate`, `/optimize`, `/walkforward`, `/multisymbol`에 rate limit 정책을 정한다.
7. 시세 데이터 공급자 후보별 가격, 라이선스, 국내/미국/크립토 범위를 비교한다.
8. 백테스트 결과와 시그널 화면에 공통 투자 고지 컴포넌트를 넣는다.
9. Playwright로 백테스트 실행, 전략 저장, 전략 비교, 시그널 등록 플로우를 테스트한다.
10. 신뢰 점수 산식 초안을 만들고 기존 결과 UI에 배지로 붙인다.

## 10. 최종 판단

이 프로젝트의 코어는 "백테스트 계산기"보다 "검증 중심 전략 실험실"에 가깝다. 이 방향을 유지하는 것이 좋다. 경쟁 서비스와 정면으로 기능 수 싸움을 하기보다, AI로 전략을 만들고, 과적합을 자동으로 경고하고, 시간이 지나도 살아남는 전략만 드러내는 제품이 되는 편이 차별성이 크다.

출시 전에는 새 기능을 많이 붙이기보다 계정, 데이터, 보안, 운영을 먼저 닫아야 한다. 그 다음 신뢰 점수, AI 전략 코치, 페이퍼 트레이딩, 시그널 자동 알림을 붙이면 제품 메시지가 분명해진다.
